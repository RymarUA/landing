# Интеграция личного кабинета с оформлением заказа

## Обзор

Реализована полная интеграция системы аутентификации с процессом оформления заказа, включающая:

1. **Автоматическое заполнение данных** из личного кабинета
2. **Умная логика email** для накладного платежа
3. **Предложение создать кабинет** на основе данных оформления

## Реализованный функционал

### 1. Автоматическое подставление данных из личного кабинета

**Что делает:**
- Если пользователь авторизован, его данные автоматически подставляются в форму оформления
- Имя и телефон заполняются из профиля пользователя
- Email НЕ подставляется для накладного платежа (требование безопасности)

**Техническая реализация:**
```typescript
// hooks/use-auth.ts
export function useAuth() {
  const [user, setUser] = useState<AuthUser | null>(null);
  // ... получение данных из /api/auth/me
}

// app/checkout/page.tsx
const { user, loading: authLoading, isAuthenticated } = useAuth();

useEffect(() => {
  if (isAuthenticated && user && !authLoading) {
    // Auto-fill name and phone if fields are empty
    if (!nameValue && user.phone) {
      setValue("name", user.phone.replace(/\D/g, ""), { shouldValidate: false });
    }
    if (!phoneValue && user.phone) {
      setValue("phone", user.phone, { shouldValidate: false });
    }
    // Note: Email is NOT auto-filled for COD payment - user must enter it manually
  }
}, [isAuthenticated, user, authLoading, nameValue, phoneValue, setValue]);
```

### 2. Логика email для накладного платежа

**Правила:**
- Email появляется в форме ТОЛЬКО при выборе накладного платежа
- Email НЕ подставляется автоматически из личного кабинета
- Пользователь должен ввести email вручную для безопасности

**Почему так сделано:**
- Безопасность: email используется для уведомлений о доставке
- Контроль пользователя: пользователь решает какой email использовать для заказов
- GDPR: отдельное согласие на использование email для коммуникаций

**Техническая реализация:**
```typescript
{/* Email field for COD */}
{paymentMethod === "cod" && (
  <div className="mt-4">
    <Field label="Email *" error={errors.email?.message}>
      <input {...register("email")} type="email" placeholder="example@mail.com" className="w-full px-4 py-3 rounded-xl border border-[#E7EFEA] text-sm focus:ring-2 focus:ring-[#C9B27C]/70 outline-none transition" />
    </Field>
  </div>
)}
```

### 3. Предложение создать личный кабинет

**Когда показывается:**
- Пользователь НЕ авторизован
- Заказ успешно оформлен (накладный платеж)
- Пользователь ввел email при оформлении

**Что предлагает:**
- Создать личный кабинет на основе введенных данных
- Сохранить данные для будущих заказов
- Получить доступ к истории заказлений

**Процесс создания:**
1. Показывается модальное окно с предложением
2. Отображаются данные, которые будут сохранены
3. Пользователь может создать кабинет или пропустить
4. При создании - регистрация через API и автоматический вход

**Техническая реализация:**
```typescript
// API endpoint
POST /api/auth/register-from-checkout
{
  "name": "Имя из формы",
  "phone": "Телефон из формы", 
  "email": "Email из формы"
}

// Ответ
{
  "success": true,
  "user": {
    "userId": "user_123",
    "phone": "+380671234567",
    "email": "user@example.com"
  }
}

// Установка JWT cookie
response.cookies.set("fhm_auth", token, {
  httpOnly: true,
  secure: process.env.NODE_ENV === "production",
  sameSite: "lax",
  maxAge: 60 * 60 * 24 * 30, // 30 days
  path: "/",
});
```

## Пользовательские сценарии

### Сценарий 1: Авторизованный пользователь
1. Пользователь входит в личный кабинет
2. Переходит в корзину → оформление
3. Имя и телефон автоматически заполняются
4. Email не заполняется (даже если есть в профиле)
5. Пользователь при необходимости редактирует данные
6. Оформляет заказ

### Сценарий 2: Неавторизованный пользователь (онлайн-оплата)
1. Пользователь заполняет все поля вручную
2. Выбирает онлайн-оплату
3. Оформляет заказ → перенаправление на WayForPay
4. Предложение создать кабинет НЕ показывается

### Сценарий 3: Неавторизованный пользователь (накладный платеж)
1. Пользователь заполняет все поля включая email
2. Выбирает накладный платеж
3. Оформляет заказ
4. Показывается предложение создать личный кабинет
5. Пользователь выбирает: создать или пропустить

## Безопасность и приватность

### Защита данных
- JWT токен с httpOnly cookie
- Валидация телефона украинского формата
- Email валидация при регистрации

### GDPR соответствие
- Email не подставляется автоматически
- Отдельное согласие на создание кабинета
- Возможность пропустить создание кабинета

## Тестирование

### Тест-кейсы:
1. Авторизованный пользователь → проверка автозаполнения
2. Неавторизованный → онлайн-оплата → нет предложения кабинета
3. Неавторизованный → накладный платеж → предложение кабинета
4. Создание кабинета → проверка JWT cookie
5. Пропуск создания → переход на страницу успеха

```javascript
// Проверка cookie

```
